# Evidence for Targeted OpenClaw Agent Review

Review only the `manualReview.items` generated by the baseline. Do not expand the work into unrelated host enumeration, active attack probes, or system modification.

## Workflow

1. Use `report-merge.mjs --review-template-out <manual-review.json>` to generate a template bound to the current baseline.
2. Perform the minimum read-only investigation required by each `requiredAction` in the template.
3. Record the conclusion in the corresponding item's `observation`. Do not change its `reviewId`, `checkId`, or `baseline` binding information.
4. Use `report-merge.mjs --manual-review <manual-review.json>` to validate and merge the evidence.
5. Inspect `agentReview` and the remaining `manualReview` in the final report.

## Observation Structure

```json
{
  "status": "PASS",
  "method": "runtime",
  "timestamp": "2026-07-16T10:00:00.000Z",
  "source": "Targeted read-only OpenClaw file-metadata check",
  "message": "The actual log file has the expected object type and permissions",
  "evidence": {
    "objectType": "file",
    "mode": "0600"
  }
}
```

Field requirements:

- `status`: Use `PASS`, `WARN`, `FAIL`, `NOT_TESTED`, `ERROR`, or `NOT_APPLICABLE`.
- `method`: Use only `static` or `runtime`.
- `timestamp`: Record the actual time when the evidence was obtained.
- `source`: Record a sanitized command category, file-metadata source, or current-version source-code contract. Do not store a full command line that contains credentials.
- `message`: Briefly describe the observed fact and reasoning behind the conclusion.
- `evidence`: Required for every observation. Store only the sanitized structured facts needed for the conclusion, never complete raw output. For `NOT_TESTED` and `ERROR`, record the checks performed, error category, or blocker.

## Status Thresholds

- `PASS`: Requires affirmative machine evidence from the current target host. A source-code contract or Agent reasoning may explain evidence but cannot prove host state by itself.
- `WARN`: Evidence shows a weak setting, ambiguous scope, or dependence on an operator-controlled exception.
- `FAIL`: Evidence directly shows that a control is missing or the configuration is insecure.
- `NOT_APPLICABLE`: Current-host configuration clearly proves that the relevant component is absent or disabled.
- `NOT_TESTED`: A targeted read-only investigation was performed, but required machine evidence is still unavailable. Record the blocker in both `message` and `evidence`.
- `ERROR`: The review command, permissions, parsing, or version contract failed.

## Security Restrictions

- Do not store tokens, passwords, API keys, cookies, Authorization values, SecretRefs, private keys, or approval socket tokens.
- Do not store private messages, model transcripts, complete environment variables, or complete configuration dumps.
- Do not create canaries, test Agents, test accounts, or temporary production configuration.
- Do not perform anonymous requests, unauthorized reads, path traversal, commands outside an allowlist, or high-risk test commands.
- Do not modify configuration, permissions, services, firewall rules, packages, users, or databases.
- The review file is part of the security-posture report. Store and share it as internal security material.
